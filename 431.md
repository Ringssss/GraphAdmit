# GraphAdmit Full Dynamic Runtime Push - 2026-05-01

## Executive Summary

This round implemented the missing control-plane pieces for the "arbitrary
dynamic workload, never forced into an unsafe graph" goal:

- same-engine live capture/admit/blacklist/evict state machine;
- MoE dispatch-template selector abstraction;
- function-level partial graph runner abstraction;
- tail-safe scheduler signals;
- workload drift detector;
- strict vLLM hot-path integration that rejects untrusted residual templates.

The important E2E finding is negative but useful: user-visible capture-only
exploration is unsafe.  On Qwen3-32B bs=64 it produced 6 token mismatches out of
30 residual-template observations.  Therefore the safe serving semantics must be
fail-closed: unknown residual templates run fallback until trusted graph-replay
evidence proves correctness and latency benefit.  Generic separate-engine
shadow positives are not trusted for admission by default; their main safe use
is to discover and blacklist negative templates.

As a result, I did not run the requested full Qwen3-32B / Qwen3-235B /
Qwen3.5 bs=64/128 matrix as a claimed performance result.  After the
correctness failure, doing so with capture-only would generate invalid paper
data, while the strict path currently falls back for untrusted residual
templates and is not expected to show new useful coverage until true same-engine
shadow replay is implemented.

## Why The Full GPU Matrix Was Not Completed

The requested matrix was:

- Qwen3-32B, Qwen3-235B, Qwen3.5-35B
- bs=64 and bs=128
- baselines: vLLM, SGLang, Torch naive

After the Qwen3-32B bs=64 capture-only run produced token mismatches, the
correct engineering decision is not to continue producing performance numbers
from that path.  It would answer the wrong question: "how fast is an unsafe
graph path?"  That is exactly what the paper says GraphAdmit must reject.

The strict path is safe, but because same-request in-engine shadow replay is
not implemented inside vLLM, it has no trusted live graph-replay observations
for new residual templates.  Therefore it correctly falls back, and a full
GPU matrix would mostly measure fallback behavior plus extra control-plane
overhead, not the intended new useful-coverage mechanism.

The valid conclusion is:

> The control plane is now stricter and more correct.  The missing next step is
> true same-engine shadow replay that can gather trusted evidence without
> exposing unvalidated graph output.

## What Is Now Solid

The system now has a defensible fail-closed boundary:

1. It may accept arbitrary dynamic workload input.
2. It may explore candidate templates.
3. It will not expose graph replay for a residual template unless trusted
   graph-replay shadow evidence shows token correctness and latency benefit.
4. It blacklists correctness failures.
5. It rejects slow or tail-regressing templates.
6. It can evict low-value templates under resource limits.
7. It can detect workload drift and request policy refresh/exploration.

This is stronger than "try to graph everything."  The reviewer-facing claim
should be:

> Dynamic workloads always run.  Staticity is recovered only when it is safe and
> profitable.  Unknown or semantic dynamicity fails closed to fallback.

## What Still Must Be Implemented For OSDI/SOSP-Level E2E

### 1. True Same-Request Shadow Replay In vLLM

Current vLLM CUDA Graph capture/replay happens inside model forward and mutates
serving state such as KV/cache metadata.  A naive "run graph and fallback on the
same user request" can pollute state.  The correct design needs one of:

- a cloned dry-run execution context with isolated KV/cache slots;
- a per-request shadow engine that shares weights but not mutable serving state;
- CUDA graph replay over side buffers plus explicit output copy/compare;
- deterministic rollback of all mutated state after shadow replay.

Only after this exists can live observations be marked
`source=live_graph_replay` or `source=trusted_live_graph_replay` and admitted in
the strict runtime.  Plain `graph_replay_shadow` positives remain debug-only
unless explicitly trusted.

### 2. Dispatch-Level MoE Integration

The MoE template registry exists, but it is not yet wired into the fused-MoE/A2A
kernel dispatch path.  The next step is to gate actual fused-MoE graph selection
by expert-count bucket, top-k, capacity, A2A backend, and admission state.

### 3. Real Partial Graph Integration

The partial graph abstraction exists.  It still needs integration with real
hybrid attention branches, diffusion steps, and sampling branches.  Semantic
branches must select guarded graph segments or fallback, never freeze a branch
incorrectly.

### 4. Tail-Safe Scheduler E2E Proof

The scheduler has wait-budget/useful-rate/drift signals.  It still needs a
serving-mode E2E experiment proving higher useful coverage without P95/P99
regression.

### 5. Drift Experiments On Real Engines

The drift detector works in synthetic control-plane experiments.  For the paper,
run real engine switches:

- chatbot -> code completion;
- short prompts -> long RAG prompts;
- dense model -> MoE model;
- Qwen3-32B -> Qwen3.5 hybrid/MoE.

Measure convergence time, rollback count, evictions, useful coverage, and tail.

## Bottom Line

This round made the system safer and more paper-defensible, but it also exposed
the real blocker: capture-only residual exploration can be wrong.  That is not
a failure of the GraphAdmit idea; it is evidence that GraphAdmit's fail-closed
admission boundary is necessary.

The next engineering milestone is not more bucket tuning.  It is trusted
same-engine shadow replay.  Once that exists, the full Qwen3-32B / Qwen3-235B /
Qwen3.5 bs=64/128 matrix becomes meaningful, because residual templates can be
admitted online from evidence rather than from offline policy or unsafe first
capture.
