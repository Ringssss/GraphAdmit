# 2026-04-30 GraphAdmit 深度代码评估与 OSDI/SOSP 论文故事

本文回答四个问题：

1. GraphAdmit 现在到底做了一件什么事，真正好在哪里。
2. 它和 vLLM CUDA Graph、SGLang CUDA Graph 的机制差异是什么。
3. 为什么有时看起来 vLLM/SGLang CUDA Graph 反而不如 Torch naive CUDA Graph。
4. 如果目标是 OSDI/SOSP，应该如何抽象成一个强而可信的系统故事。

结论先放前面：

> GraphAdmit 最强的点不是“任意动态都能 graph”，而是把动态 LLM serving 中原本靠经验处理的 CUDA Graph 静态性问题变成一个 fail-closed 的运行时决策问题：系统观察动态性，恢复可恢复的静态性，只在 token-correct 且实测快于 fallback 的模板上 replay，其他请求永远 fallback。

这比“支持任意动态进入 CUDA Graph”更可信。OSDI/SOSP 的正确包装应该是：

> **Validated staticity recovery for dynamic LLM serving.**

或者更论文一点：

> **GraphAdmit: Fail-Closed Staticity Recovery for Dynamic LLM Serving.**

## 1. 当前系统真正好在哪里

GraphAdmit 的核心价值不是多加几个 bucket，也不是比 vLLM/SGLang 多 capture 几张图。它的核心价值是引入了一个新的 correctness/performance contract：

> 动态 workload 总能进系统；CUDA Graph 只是被系统有条件地 admit。不能证明正确或不能证明更快，就回到 fallback。

这件事在系统上有三个实际好处。

### 1.1 把 CUDA Graph 的目标从 hit rate 改成 useful coverage

vLLM 和 SGLang 的默认 graph 机制大多是在回答：

> 这个 batch 能不能命中某个已 capture 的 shape/key？

GraphAdmit 改成回答：

> 这个 batch 命中的 graph path 是否 token-correct，并且是否比 fallback 更快？

这就是 useful coverage：

```text
useful_coverage = token_correct && graph_latency < fallback_latency
```

这个指标比 graph hit rate 更适合动态 serving。已有结果说明 graph hit 本身没有意义：

- dInfer unsafe graph: 快但 token 错，useful coverage 是 0。
- long over-capture: token 正确但比 fallback 慢，useful coverage 是 0。
- aggressive arena: graph 能跑但 short/outlier 回退严重，negative graph rate 高。

GraphAdmit 的真正强点是把这些 bad graph path 排除掉。

### 1.2 把“动态性”拆开处理，而不是把所有动态都当 shape miss

动态 LLM serving 的动态性不是单一的 `num_tokens`：

- shape dynamicity: token 数、batch size、sequence length、expert token count。
- metadata address dynamicity: `positions`、`slot_mapping`、`query_start_loc`、`seq_lens`、`block_table` 的地址和 view。
- metadata value dynamicity: 同一个地址里每次写入不同语义值。
- function/control dynamicity: hybrid attention、sliding/full attention branch、MoE routing、sampling branch、diffusion mask/update step。
- workload drift: chatbot、code completion、long-prefill、mixed prefill-decode 的分布变化。

GraphAdmit 的设计区别是：先分类，再 canonicalize，再执行。不能把 semantic dynamic 冻住，只能把 representational/address dynamic 放进 arena 或模板化。

这点很适合作为论文贡献，因为它不是简单工程调参，而是给 CUDA Graph serving 提供了一个可解释的动态性模型。

### 1.3 它是 fail-closed，而不是 best-effort graph

代码里多个地方都体现了 fail-closed：

- `prefill_graph/runtime/admission.py`
  - `OnlineSelfLearningAdmissionController` 以 correctness failure、useful rate、EWMA saving、P95 regression 为 admission 条件。
  - correctness failure 会 blacklist template。
  - latency saving 不够时 fallback。
- `prefill_graph/runtime/residual_capture.py`
  - residual candidate 必须有 exact-template evidence。
  - 需要通过 useful-rate、average saving、P95 saving、P95 regression、max regression guard。
  - tail template 可以用更严格阈值。
- `prefill_graph/runtime/arena.py`
  - arena 只是固定地址和 padding，不隐藏 semantic value。
  - MoE `expert_ids`、`expert_counts` 标成 semantic_value，不应被错误冻结。
- `external/FlowPrefill/vllm/v1/cudagraph_dispatcher.py`
  - vLLM patch 在 dispatch 前先读 runtime policy。
  - extra templates 受 template budget、multi-request metadata guard、live admission guard 控制。
  - default/cp action 超过 base capture size 会 fallback，避免误用 ours-only extra graph。
- `external/FlowPrefill/vllm/v1/worker/gpu_model_runner.py`
  - fixed metadata arena 只在 policy window 内启用。
  - request-axis metadata padding 后填充 inactive/padded region。
  - `slot_mapping` padded token region 填 `-1`，避免无效 token 写 KV。

这让系统的安全边界比“尽可能 replay”更强。

## 2. 当前代码实现的真实边界

### 2.1 已经比较扎实的部分

**Runtime policy and admission**

- `prefill_graph/runtime/admission.py`
- `graphadmit/policy.py`
- `graphadmit/cli.py`
- `external/FlowPrefill/vllm/v1/cudagraph_dispatcher.py`

已经有 standalone admission controller，也有 vLLM hot-path policy consumption。vLLM dispatcher 能 ingest live observation file，根据 samples/useful-rate/saving/tail guard 决定 graph or fallback。

需要诚实说明：当前 vLLM 的 capture sizes 仍然在 engine 初始化期决定，所以现在更准确地说是 **online admission / next-round residual capture**，不是完整的 same-engine dynamic capture/evict。

**Fixed-address metadata arena and key collapse**

- `prefill_graph/runtime/arena.py`
- `external/FlowPrefill/vllm/v1/worker/gpu_model_runner.py`
- `external/FlowPrefill/vllm/v1/cudagraph_dispatcher.py`

已经覆盖 request-axis metadata：`query_start_loc`、`seq_lens`、`request_indices`、`is_prefilling`、`block_table`。token-axis `positions`、`slot_mapping`、`token_active_mask` 也有 abstraction 和 vLLM integration 迹象。

Controlled key-collapse 结果 `results/kc_qwen32_fixed_arena_compare.json` 证明 same total tokens / different num_reqs/layout 可以在 fixed-address arena 下减少 key variants，avg key-variant reduction 是 `1.50x`。

**Residual capture / PCG-tail**

- `prefill_graph/runtime/residual_capture.py`
- `benchmarks/plan_residual_capture_policy.py`
- `benchmarks/vllm_flowprefill_workload.py`

已经支持 SGLang-style bucket preset、exact evidence、tail-aware thresholds、interval DP selection。最新结果显示：

- Qwen3-235B: GraphAdmit PCG-tail `55.48 ms` avg，SGLang PCG `58.37 ms` avg，vLLM max512 CP `141.29 ms` avg。
- Qwen3-32B: GraphAdmit PCG-tail `38.99 ms` avg，SGLang PCG `43.29 ms` avg，vLLM max512 CP `43.37 ms` avg。

**dInfer / dynamic function correctness**

- `benchmarks/dinfer_qwentrace_offline.py`
- `benchmarks/llada2_dinfer_kvcache_graph_generate.py`

dInfer/LLaDA 结果比较强：runtime graph admission `83.94s` vs eager `143.37s`，约 `1.708x`，token-correct。这个方向证明了“动态 function 不应 blind graph，必须 correctness-admitted”。

**Package and artifact**

- `graphadmit/`
- `pyproject.toml`
- `README.md`
- `summary.md`
- `patches/vllm_staticity.patch`

已经变成 `pip install git+https://github.com/Ringssss/GraphAdmit.git` 可安装的工具形态，有 `graphadmit make-policy`、`graphadmit vllm-env`、`graphadmit vllm-patch` 等 CLI。

### 2.2 还没有完全闭环的部分

这些必须在 OSDI/SOSP 论文里处理，不能假装已经全部完成。

**In-engine live capture/evict 还不完整**

当前 hot path 能做 live admission 和 observation ingestion，但 vLLM CUDA Graph capture family 仍固定在 engine 初始化。要写强 claim，需要做到：

- engine 内发现 residual miss；
- shadow fallback timing；
- token correctness validation；
- capture new template；
- admit/blacklist；
- evict low-ROI template；
- workload drift 后重新收敛。

**MoE metadata templates 还没有接管 fused-MoE/all-to-all dispatch**

`ExpertMetadataCanonicalizer` 和 MoE profile 已经有，但目前更多是 metadata extraction/template abstraction。OSDI/SOSP 级别需要把它变成 dispatch-level decision：

- expert capacity bucket；
- expert imbalance guard；
- fused-MoE kernel selection；
- all-to-all backend fallback；
- per-expert token count template；
- rare imbalance fallback。

**Function-level partial graph 还主要是 abstraction**

`prefill_graph/runtime/partial_graph.py` 设计是对的：attention mode、MoE capacity、diffusion step 都注册 guarded segment。但还没有深度接入真实 hybrid attention / sliding-full attention / diffusion branch 的执行图切分。

**Scheduler 还没有强 tail 证据**

`TemplateAwareScheduler` 和 `SlaAwareTemplateScheduler` 已经实现，vLLM scheduler 也有 hook，但当前结果不能支撑“scheduler 主动制造静态性且 tail 不退化”的强 claim。Qwen3-32B 的 P95/P99 仍略弱于 SGLang。

**Qwen3.5 whole-run token identity 有 caveat**

Qwen3.5 hybrid/MoE 的 extra-template observations 是 token-correct 的，但 whole-run across all modes 有小路径 nondeterminism。论文里应该写：

> admitted GraphAdmit templates are token-correct against shadow fallback.

不要写：

> all Qwen3.5 modes are whole-run token-identical.

## 3. 和 vLLM CUDA Graph 的核心区别

vLLM 的 dispatcher 本质是：

```text
runtime num_tokens/num_reqs/layout
  -> padded BatchDescriptor
  -> lookup precomputed cudagraph_keys
  -> FULL / PIECEWISE / NONE
```

它的强项是和 vLLM runtime 深度集成，能处理 FULL、PIECEWISE、decode-only、LoRA key、padding key 等复杂情况。弱点是它的 dispatch key 主要表达“能否 replay”，而不是“是否值得 replay”。

GraphAdmit 在 vLLM 里加的是 dispatch 前后的控制面：

```text
runtime batch
  -> runtime policy range
  -> live admission stats
  -> correctness/latency/template-budget guards
  -> fixed metadata arena gate
  -> key collapse only inside admitted window
  -> graph or fallback
```

最关键的差别：

| 维度 | vLLM CG | GraphAdmit on vLLM |
|---|---|---|
| 目标 | graph key hit | useful coverage |
| 模板来源 | 初始化时固定 capture sizes | fixed capture + residual/admitted templates |
| 是否知道 replay 更慢 | 默认不知道 | latency guard 拒绝 |
| 是否知道 replay token 错 | 默认不知道 | token-correct guard 拒绝 |
| metadata address 动态 | 进入 key 或导致 miss | fixed-address arena |
| same total tokens different layout | 容易变多 key | arena-gated key collapse |
| multi-request extra graph | 风险较高 | 需要 metadata arena / explicit allow |
| workload drift | 静态配置 | online stats / next-round refresh |

所以我们比 vLLM 好，不是因为 vLLM CUDA Graph 做得差，而是因为 vLLM 没有把 graph replay 当成一个需要在线验证的 admission problem。

## 4. 和 SGLang CUDA Graph 的核心区别

SGLang 是非常强的 baseline，不能当 strawman。根据本地源码 review：

- decode CUDA Graph:
  - `sglang/srt/model_executor/cuda_graph_runner.py`
  - capture batch-size buckets；
  - replay 时 pad 到最近 capture batch size；
  - 用 persistent input buffers 固定地址。
- Piecewise CUDA Graph:
  - `sglang/srt/model_executor/piecewise_cuda_graph_runner.py`
  - `sglang/srt/compilation/cuda_piecewise_backend.py`
  - capture token-count buckets；
  - forward 中每个 piece/layer 通过 compile backend 捕获；
  - runtime token 数 pad 到最近 captured token size。
- bucket 生成：
  - `server_args.py` 生成 `4,8,...,32`、`48..256`、`288..512`、`576..1024`、`1280..4096` 等 PCG token buckets。
- auto-disable 条件很多：
  - MoE A2A backend；
  - LoRA；
  - multimodal/VLM；
  - diffusion LLM；
  - deterministic inference；
  - disaggregation；
  - pipeline parallel；
  - DP attention；
  - non-CUDA；
  - expert distribution recorder 等。

SGLang PCG 的强项是工程非常深：它已经实现了 token-piecewise CUDA Graph、padding、per-piece capture、attention/MoE 相关 bypass。它在 Qwen3-235B 上一度明显强于旧 GraphAdmit，原因就是它覆盖了长 prefill token buckets。

GraphAdmit 和它的区别不是“有没有 PCG”。区别是：

| 维度 | SGLang PCG | GraphAdmit |
|---|---|---|
| 模板族 | 预设 broad PCG buckets | candidate buckets + useful admission |
| admission | bucket 存在即可尝试 replay | token-correct + latency-profitable 才 admit |
| bad graph 处理 | 靠 fallback/disable 条件 | template blacklist / tail guard / negative graph rejection |
| dynamicity taxonomy | 工程规则为主 | semantic / representational / address / control 分类 |
| generalized framework | SGLang runtime 内强实现 | vLLM/dInfer/Torch 通用 admission substrate |
| 兼容边界 | PCG 多条件 auto-disable | 目标是 partial graph + guarded fallback |

当前结果应这样写：

- 对 vLLM：GraphAdmit 已经有强而稳定的优势。
- 对 SGLang：GraphAdmit 不是全面碾压，但能在若干动态场景匹配或超过 SGLang PCG，并提供更强的 correctness/latency safety boundary。

具体数据：

- Qwen3.5-35B hybrid/MoE:
  - GraphAdmit live runtime `42.82 ms` avg。
  - SGLang default CG `116.47 ms` avg。
  - SGLang piecewise CG `286.58 ms` avg。
  - 这里 SGLang 日志显示 hybrid/mamba/no-buffer 相关限制削弱 overlap/PCG 路径。
- Qwen3-235B:
  - GraphAdmit PCG-tail `55.48 ms` avg。
  - SGLang PCG `58.37 ms` avg。
  - vLLM max512 CP `141.29 ms` avg。
- Qwen3-32B:
  - GraphAdmit PCG-tail `38.99 ms` avg。
  - SGLang PCG `43.29 ms` avg。
  - 但 SGLang P95/P99 更好：`111.27/123.17 ms` vs GraphAdmit `115.85/127.24 ms`。

论文里不要写“GraphAdmit dominates SGLang”。应该写：

> GraphAdmit reaches SGLang-level PCG performance on strong baselines while adding fail-closed admission and a broader dynamicity model. On workloads where broad PCG is disabled, weakened, or unprofitable, GraphAdmit can recover safe templates that static PCG policies should not blindly admit.

## 5. 为什么 vLLM/SGLang CG 有时看起来不如 Torch naive

这件事看起来奇怪，主要是因为比较对象被混在了一起。

### 5.1 Torch microbench 里的强结果不是 naive CUDA Graph

`results/torch_naive_cudagraph_compositional_microbench_qwen35_context.json` 里真正的 naive baseline 是：

| Plan | Speedup | 解释 |
|---|---:|---|
| `exact_only_graph_32_64` | `1.10x` | 只 replay exact captured 32/64，其他 fallback |
| `pad_to_next_template` | `1.88x` | padding 到模板，不是 naive |
| `pack_adjacent_requests` | `3.56x` | 把相邻 token-local requests 组合成一个 graph，不是 naive |
| `residual_learn_then_replay` | `1.80x` | residual capture/admission，不是 naive |

所以“Torch naive 比 SGLang/vLLM 强”这个说法是不准确的。真正 naive Torch exact-only 只有 `1.10x`。

强的 `pack_adjacent_requests` 是我们的 compositional replay idea。它在 token-local MLP 上合法，因为 token 之间没有 attention/KV/MoE 依赖。真实 LLM prefill 里不能直接把 `55+41` 合成 `96`，除非你同时证明：

- attention mask 不跨 request 泄漏；
- position/slot_mapping/block_table 正确；
- sampling/logits slicing 正确；
- MoE routing 不被合并改变；
- KV write 不污染；
- output de-pack token-correct。

这正是 GraphAdmit 的 arena、mask、metadata contract 要解决的问题。

### 5.2 vLLM/SGLang 是完整 serving stack，Torch microbench 不是

vLLM/SGLang E2E 里面包含：

- tokenizer/prompt construction；
- scheduler；
- KV cache allocation；
- attention backend metadata；
- tensor parallel communication；
- MoE routing/all-to-all；
- logits/sampling；
- graph capture memory pool；
- backend-specific padding；
- Python/server orchestration。

Torch microbench 只是 token-local MLP：

```text
x -> LayerNorm -> Linear/Gate/Down -> x
```

它没有 attention、没有 KV cache、没有 MoE dispatch、没有 scheduler，也没有多卡通信。它适合证明机制潜力，不适合当 serving baseline。

### 5.3 full graph 可能更慢是正常现象

CUDA Graph 省的是 launch/CPU overhead，不保证 GPU compute 更少。下面情况会让 graph replay 更慢：

- padding waste 大，静态模板跑了很多无效 token。
- graph capture 选择了不适合当前 token/layout 的 kernel。
- metadata copy-in 成本超过 launch saving。
- graph-hostile op、communication、custom all-reduce、MoE all-to-all 进入图后失去原有 overlap。
- full graph 把 dynamic branch 固定住，导致 fallback 之外的路径更差。
- broad PCG capture 很多模板，init/memory pressure 变高，影响后续 allocator/cache 行为。

所以 Figure 1 用“static CUDA graph replay is not always useful”非常合理，但注意不要暗示 SGLang/vLLM 都是 naive。正确动机是：

> Existing CUDA Graph policies conflate replayability with usefulness. In dynamic LLM serving, a graph path can be replayable but slower, or fast but semantically wrong.

## 6. OSDI/SOSP 的三大痛点

论文最好不要从“我们比某个框架快”开始，而从系统痛点开始。

### 痛点 1: Graph hit rate 不是正确目标

现有系统默认倾向于问“能不能 replay”。但动态 LLM serving 里 graph path 有三种坏情况：

1. 不能 replay，shape/key miss。
2. 能 replay但更慢，negative graph。
3. 能 replay且更快但 token 错，unsafe graph。

所以 graph hit rate 会误导系统。真正目标应该是 useful coverage。

### 痛点 2: 动态性来源异构，不能一刀切静态化

动态性来自 shape、metadata address、metadata values、function branches、MoE routing、workload drift。不同动态性需要不同处理：

- representational/address dynamic 可以 canonicalize；
- semantic dynamic 只能保留为 runtime value；
- control dynamic 需要 guarded partial graph；
- workload drift 需要 online admission/eviction。

现有 engine 的 bucket/padding 策略不足以表达这个边界。

### 痛点 3: workload 会变，离线 graph policy 不可靠

离线 calibration 在新 workload 或 workload drift 下容易失效。Reviewer 会问：

- 新 workload 来了能否自动收敛？
- chatbot 切到 code completion 多久 adapt？
- 长 prefill 混入短 decode 是否 tail regression？
- graph memory 不够时谁 evict？

这要求 engine 内部有 explore / validate / admit / blacklist / evict 的闭环。

## 7. OSDI/SOSP 的三大挑战

### 挑战 1: 如何知道哪些动态可以安全恢复

系统必须区分：

- semantic dynamic: 不能冻结。
- representational dynamic: 可以转换成等价静态表示。
- address dynamic: 可以固定 buffer 地址，动态 copy-in。
- control dynamic: 需要 guard 和 partial fallback。

没有这个分类，系统会把 semantic branch 错误 capture 成 static path。

### 挑战 2: 如何恢复静态性而不改变语义

CUDA Graph 要求固定地址、shape 和 dispatch key。LLM serving 的 metadata 每次都变。系统要做到：

- fixed-address arena；
- token-axis padding；
- inactive mask；
- request-axis canonicalization；
- key collapse 只在 arena 证明安全时启用；
- MoE expert capacity template；
- function-level partial graph。

这不是简单 padding，因为 padding 必须和 attention/KV/MoE semantics 一起验证。

### 挑战 3: 如何在 workload drift 下只执行有收益的 graph

即使 graph 正确，也可能不快。系统必须在线维护：

- correctness evidence；
- graph/fallback paired timing；
- capture/warmup amortization；
- useful-rate；
- tail regression；
- blacklist；
- eviction；
- scheduler wait ROI。

这让 graph execution 从 static policy 变成 runtime control problem。

## 8. 三个组件如何映射挑战

建议论文只讲三个组件，不讲 5 个 box。Admission 和 scheduler 都收编进 Execute。

| 组件 | 解决的挑战 | 当前代码 |
|---|---|---|
| **Classify: Dynamicity Contract** | 识别 shape/metadata/function/MoE/workload dynamicity，区分 semantic vs recoverable | profiler/analyzer, `analyze_*`, vLLM trace JSONL, MoE profile |
| **Canonicalize: Staticity Recovery Substrate** | 用 arena/padding/key-collapse/MoE metadata/partial graph template 恢复可恢复静态性 | `arena.py`, vLLM fixed metadata arena, residual templates |
| **Execute: Validated Template Execution** | online admission、shadow validation、fallback、blacklist、evict、template-aware dispatch | `admission.py`, `residual_capture.py`, `scheduler.py`, vLLM dispatcher patch |

一句话 overview 可以这样写：

> GraphAdmit classifies each source of dynamicity, canonicalizes recoverable dynamics into address-stable templates, and executes templates only when online admission proves them token-correct and faster than fallback.

这三个动词非常适合贯穿 abstract、intro、architecture。

## 9. 论文贡献可以怎么写

### Contribution 1: Dynamicity contract for graph serving

提出一种 runtime contract，把动态性分成 semantic、representational、address、control、drift，并说明哪些能被 CUDA Graph staticity recovery 安全处理。

这部分的 selling point 是“正确性边界”。Reviewer 会喜欢，因为它不是只说快，而是说什么时候不能 graph。

### Contribution 2: Staticity recovery substrate

提出 fixed-address metadata arena、token-axis canonicalization、arena-gated key collapse、MoE/function template abstraction。

这部分的 selling point 是“把 layout/address dynamic 从 graph key 里移走，但不冻结 semantic value”。

### Contribution 3: Validated template execution

提出 online useful-coverage admission：template 只有在 token-correct、faster-than-fallback、tail-safe、amortized-positive 时才进入 graph path。

这部分的 selling point 是“workload 变化时 fail-closed 收敛”，而不是离线 oracle。

## 10. 实验结果该如何讲

### 10.1 Figure 1: CUDA Graph replay is not always useful

Figure 1 应该展示当前 CUDA Graph 的痛点，不要提前宣传 GraphAdmit。推荐只放 baseline 行为：

- 横轴：request index 或 token count。
- 纵轴：speedup over fallback 或 latency ratio。
- fallback parity line。
- 蓝色：graph replay faster than fallback。
- 橙色：graph replay slower than fallback。
- 标注：`21/56 graph replays are slower` 这种动机事实。

这张图要表达：

> Static CUDA Graph replay can be replayable but unprofitable.

不要在 Figure 1 里写 GraphAdmit 优势，避免动机图变成结果图。

### 10.2 vLLM 结果

vLLM 是最适合主打的 baseline，因为 GraphAdmit 的机制差异最明显。

Qwen3.5 hybrid/MoE:

- vLLM max512 CP: `254.59 ms` avg。
- GraphAdmit live runtime: `42.82 ms` avg。
- 平均 `5.95x`。
- P95: `759.81 ms` -> `56.68 ms`。

Qwen3-235B:

- vLLM max512 CP: `141.29 ms` avg。
- GraphAdmit PCG-tail: `55.48 ms` avg。
- 平均 `2.55x`。

Qwen3-32B:

- vLLM max512 CP: `43.37 ms` avg。
- GraphAdmit PCG-tail: `38.99 ms` avg。
- 平均 `1.11x`。

对 vLLM 的结论：

> GraphAdmit improves useful coverage beyond vLLM's fixed capture family, especially for dynamic prefill/MoE/hybrid workloads.

### 10.3 SGLang 结果

SGLang 应该作为 strongest baseline，不要轻视。

可以写：

- Qwen3.5 hybrid/MoE: GraphAdmit 明显快于 SGLang default/PCG，因为 SGLang 在该模型上的 PCG/overlap 路径受限。
- Qwen3-235B: GraphAdmit PCG-tail 略快于 SGLang PCG。
- Qwen3-32B: GraphAdmit 平均更快，但 tail 略差。

这能支撑：

> GraphAdmit can match or exceed SGLang PCG on selected dynamic workloads while adding fail-closed admission.

不能支撑：

> GraphAdmit universally dominates SGLang.

### 10.4 dInfer/diffusion 结果

dInfer 是“dynamic function correctness”的关键证据：

- eager: `143.37s`。
- runtime graph admission: `83.94s`。
- speedup `1.708x`。
- token-correct。

它说明 CUDA Graph 在动态 function 场景里最大问题不是 shape，而是 semantic correctness。GraphAdmit 的 admission/validation 能把 unsafe fast graph 变成 safe useful graph。

### 10.5 Ablation 必须包括

建议 OSDI/SOSP 评估至少有这些 ablation：

| Ablation | 要证明什么 |
|---|---|
| no admission | graph hit 高但 negative graph 多 |
| no arena | key explosion / metadata miss |
| no token-axis padding | long prefill miss / fallback |
| no residual capture | workload drift 下无法覆盖新模板 |
| no tail guard | P95/P99 regression |
| no scheduler | admitted templates hit rate 下降 |
| SGLang bucket family without admission | broad coverage 但 bad template 不会被拒绝 |

## 11. 离 OSDI/SOSP 强版本还差什么

如果目标是顶会，不应只把当前系统包装得更好。还需要把下面几个 gap 补成硬结果。

### 11.1 真正 in-engine live admission

当前最容易被 reviewer 打的问题是“online 还不够 online”。要补：

- same engine 内探索候选模板；
- paired fallback timing；
- correctness shadow validation；
- live admit；
- live blacklist；
- live eviction；
- drift adaptation time。

评估要给：

- cold workload 收敛到 stable policy 的请求数；
- workload A 切到 workload B 后多久恢复 useful coverage；
- bad template 被发现和 blacklisted 的请求数；
- eviction 是否降低 graph memory。

### 11.2 长 prefill token-axis arena 要做到 SGLang PCG 等价覆盖

Qwen3-235B 早期输给 SGLang 的根因就是长请求 `2829/3023` token 没覆盖。PCG-tail 加 `3072` 后追上了。

OSDI/SOSP 版要系统化：

- token-axis arena for `positions/slot_mapping/block_table/query_start_loc`；
- templates: 768/832/896/1024/1536/2048/3072/4096；
- admit only if measured useful；
- report fewer admitted templates vs SGLang broad PCG。

### 11.3 MoE dispatch-level templates

现在 MoE profile 很丰富，但还不够“系统”。需要把 expert metadata 接入 fused-MoE/all-to-all dispatch：

- expert-count capacity bucket；
- expert active mask；
- token permutation arena；
- imbalance fallback；
- A2A backend guard；
- MoE layer grouping。

这会让 Qwen3-235B/Qwen3.5 的故事更硬。

### 11.4 Function-level partial graph 要接真实 branch

把 `partial_graph.py` 从 abstraction 变成真实路径：

- hybrid attention: sliding/full attention branch；
- diffusion: mask/update step；
- sampling: branch outside graph；
- MoE: graph dense blocks, fallback routing/communication。

这能支撑“任意 dynamic function 都不会迫使系统错误 graph 化”的 claim。

### 11.5 Scheduler 必须证明 tail-safe

模板感知 scheduler 不能只提高 hit rate。必须证明：

- bounded wait 小于 expected saving；
- admitted template 才等待；
- P95/P99 不退化；
- under load 和 low load 都安全；
- starvation 不发生。

否则 scheduler 只能作为 optional optimization，不能作为核心 claim。

### 11.6 Artifact 要更像生产系统

现在 package 已经可安装，但 OSDI/SOSP artifact 最好再补：

- clean vLLM fork or patch against exact upstream commit；
- one-command reproduction；
- policy log schema；
- graph memory accounting；
- template dump/inspection CLI；
- live admission dashboard or JSON summary；
- failure injection tests。

## 12. 如何回答 reviewer 的关键问题

**Q1: 你们是不是只是调 bucket？**

不是。Bucket 只是 candidate template family。核心是 runtime contract：metadata arena、key collapse、correctness/latency admission、fallback/blacklist。证据是 bad graph cases 被拒绝，same-token different-layout key collapse 安全，dInfer unsafe graph 被 correctness guard 排除。

**Q2: SGLang 已经有 PCG，你们新在哪里？**

SGLang PCG 是强 baseline，但它主要是预设 token-piecewise capture family。GraphAdmit 的新点是 fail-closed useful admission 和动态性分类：graph path 只有被证明正确且更快才 admit；不可恢复的 semantic/control dynamic 走 guarded fallback。这是一个安全边界和泛化框架，不是单一 runtime 的 bucket list。

**Q3: 你们 online 吗？**

当前代码已经有 hot-path live admission 和 observation ingestion，但 capture family 仍受 vLLM 初始化限制。论文强版本必须补 same-engine live capture/evict。当前可诚实写为 online admission / next-round residual capture；OSDI/SOSP 版应升级为 full in-engine explore/validate/admit/evict。

**Q4: 会不会错误冻结 semantic dynamic？**

设计上不允许。Semantic fields 如 MoE expert ids/counts、diffusion mask/update、attention mode 进入 guard 或 runtime value，不被 key collapse 消掉。错误或不确定时 fallback。

**Q5: 为什么不直接用 SGLang PCG bucket family？**

可以用，而且 GraphAdmit 已经支持 SGLang PCG bucket preset。区别是 GraphAdmit 不会因为 bucket 存在就长期 admit，而是用 measured useful coverage 和 tail guard 选择少量高收益模板。

## 13. 最好的论文故事线

### Introduction 结构

1. CUDA Graph 是降低 LLM serving launch overhead 的重要技术，但它假设静态 execution。
2. 动态 serving 违反静态性：shape、metadata address、routing、function branch、workload drift。
3. 现有系统用 fixed buckets、padding、piecewise graph 缓解，但优化目标仍接近 graph hit rate。
4. 我们发现 graph replay 可 replay 但更慢，甚至可 replay 且更快但 token 错。
5. 因此需要一个新的目标：useful coverage。
6. GraphAdmit 通过 classify/canonicalize/execute 三阶段实现 fail-closed staticity recovery。
7. 结果：vLLM 上显著提升，SGLang-level 性能且更安全，dInfer dynamic function token-correct 加速。

### Architecture 结构

```text
Incoming workload
  -> Classify dynamicity
       staticity contract
  -> Canonicalize recoverable dynamics
       fixed metadata arena
       token-axis arena
       MoE/function templates
  -> Execute validated templates
       dispatch
       online admission
       partial graph/fallback
       scheduler
```

### Title 候选

- GraphAdmit: Fail-Closed Staticity Recovery for Dynamic LLM Serving
- GraphAdmit: Validated CUDA Graph Execution for Dynamic LLM Serving
- Recovering Staticity in Dynamic LLM Serving

我最推荐第一个，因为它把“不是任意 graph，而是 fail-closed”讲清楚。

## 14. 当前最准确的一句话 claim

> GraphAdmit makes CUDA Graph practical for dynamic LLM serving by classifying dynamicity, canonicalizing recoverable metadata/address dynamics into graph-stable templates, and executing templates only when online admission proves them token-correct and faster than fallback. Dynamic workloads always run; staticity is recovered only when safe and profitable.

这句话比“支持任意动态 CUDA Graph”更强，也更不容易被 reviewer 攻破。

## 16. Bottom line

GraphAdmit 当前已经足够形成一篇强系统论文的核心：

- 它指出 graph hit rate 不是动态 serving 的正确目标。
- 它提出 useful coverage 和 fail-closed admission。
- 它把 shape/address/metadata/function/workload drift 分层处理。
- 它已经在 vLLM、dInfer、Torch microbench、SGLang comparison 上有实验证据。

但 OSDI/SOSP 版必须继续补强：

- same-engine live capture/evict；
- MoE dispatch-level template；
- real partial graph integration；
- tail-safe scheduler；
- broader workloads and drift experiments；
- full artifact reproducibility。

如果这些补上，论文故事会很完整：

> 现有 CUDA Graph serving runtime 追求 replayability；GraphAdmit 追求 validated usefulness。它允许任意动态 workload 进入系统，但永远不会被迫错误 graph 化。
