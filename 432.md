# GraphAdmit Fail-Closed Runtime Status - 2026-05-01

## Bottom Line

This round makes the system substantially more solid, but the honest conclusion
is not "every dynamic request can be graphed."  The implemented and tested
property is stronger and safer:

> Arbitrary dynamic workloads may enter the engine.  The runtime will only use a
> graph path when the template has trusted evidence that replay is token-correct
> and faster than fallback.  Unknown, slow, or semantically unsafe dynamicity
> fails closed to fallback.

This is the right reviewer-facing boundary.  It directly addresses the earlier
weakness that online admission could be interpreted as an offline/oracle policy.

## What Was Fixed

### 1. Shadow positives no longer over-admit

The dispatcher no longer treats generic `graph_replay_shadow` positives as
trusted admission evidence.  By default:

- positive admission requires `source=live_graph_replay` or
  `source=trusted_live_graph_replay`;
- `graph_replay_shadow` positives require the explicit debug knob
  `STATICITY_VLLM_TRUST_SHADOW_POSITIVE_OBSERVATIONS=1`;
- `graph_replay_shadow` negatives are still consumed and blacklist unsafe
  templates;
- positive alias expansion is off by default, so exact-token failures do not
  get hidden by broad bucket aliases.

This prevents a separate-engine or offline shadow run from admitting a template
inside the target engine.

### 2. Fallback metadata is no longer canonicalized by graph-only arenas

`gpu_model_runner.py` now gates fixed metadata arena use on the active
`CUDAGraphMode`.  If the dispatcher rejects a residual graph template and sends
the request to fallback, `positions`, `slot_mapping`, `query_start_loc`, and
`block_table` do not get forced through the graph arena.

The earlier one-token mismatch investigation showed why this matters: fallback
must remain the semantic baseline, not a partially graph-shaped path.

### 3. UBatch FULL replay is guarded

`gpu_ubatch_wrapper.py` now checks the same replay guard before FULL ubatch
graph replay.  If a live-capture-only descriptor is blocked, it emits an
`ubatch_replay_blocked_fallback` trace event and runs fallback instead.

This closes a bypass where a graph could be replayed below the main dispatcher
decision.

### 4. Exact template ids and prefix-cache control were added

The benchmark harness can now write exact template ids such as:

```text
ours_cp:734:758:tokens=756:template=768:reqs=1
```

This avoids admitting the whole `768` bucket because one nearby token count was
safe.  The harness also has `--disable-prefix-caching`, which gives cleaner
cross-engine correctness comparisons when the same prompt appears multiple
times in a trace.

### 5. Packaging now matches the patched source tree

The installable package now ships the updated vLLM patch and manifest:

- `patches/vllm_staticity.patch`
- `patches/vllm_staticity.patch.manifest.json`
- `graphadmit/resources/vllm_staticity.patch`
- `graphadmit/resources/vllm_staticity.patch.manifest.json`

The patch manifest covers 8 files, including the previously missing
`vllm/_cg_trace.py` and the new `vllm/v1/worker/gpu_ubatch_wrapper.py` guard.

Result file:

```text

```

| Config | Avg ms | P95 ms | P99 ms | Token match |
|---|---:|---:|---:|---:|
| vLLM max512 CP FULL_AND_PIECEWISE | 43.42 | 113.96 | 127.99 | 64/64 |
| Shadow-engine runtime | 59.61 | 242.30 | 301.76 | 59/64 |
| Strict single-engine runtime | 43.61 | 113.27 | 126.76 | 64/64 |

Interpretation:

- the shadow path deliberately exposes unsafe candidates and produced
  mismatches;
- the strict path consumed the negative evidence and did not admit unsafe
  residual graph replay;
- strict runtime is latency-parity with vLLM CP because unsafe residual
  templates fall back.

### bs=128 strict fail-closed, prefix cache disabled

Result file:

```text

```

| Config | Avg ms | P50 ms | P95 ms | P99 ms | Token match |
|---|---:|---:|---:|---:|---:|
| vLLM max512 CP FULL_AND_PIECEWISE | 40.48 | 36.92 | 111.97 | 125.26 | 128/128 |
| Shadow-engine runtime | 48.17 | 37.85 | 127.86 | 196.99 | 122/128 |
| Strict single-engine runtime | 41.10 | 37.07 | 111.35 | 125.09 | 128/128 |

The earlier prefix-cache-enabled run had 1/128 mismatch in strict mode.  The
dispatcher trace showed that request ran with `mode=NONE` and no metadata arena,
so the mismatch was not caused by graph replay.  Disabling prefix cache removed
that cross-engine nondeterminism and produced 128/128 token identity.

Dispatcher evidence for the no-prefix-cache run:

- strict runtime dispatches: `752 NONE`, `412 PIECEWISE`, `212 FULL`
- residual rejects include:
  `staticity_runtime_action_ours_cp_live_correctness_failure`
- attention arena in strict runtime:
  `932` non-arena events, `8` arena events, all `for_cudagraph_capture=True`

So fallback requests are not being silently arena-canonicalized.

## What Is Now Actually Solid

The system now has a defensible three-layer story:

1. **Classify dynamicity.**
   Shape, metadata address, function/control branch, MoE routing, and workload
   drift are surfaced as distinct causes rather than collapsed into "dynamic
   shape."

2. **Canonicalize recoverable dynamicity.**
   Token-axis buckets, fixed-address metadata arenas, arena-gated key collapse,
   MoE metadata templates, and residual candidate templates produce graph-stable
   candidates.

3. **Execute only with admission and guards.**
   The dispatcher, ubatch wrapper, partial graph manager, MoE dispatch selector,
   scheduler signals, live-capture manager, blacklist, and drift detector all
   enforce fail-closed execution.

This is enough to claim:

> GraphAdmit lets arbitrary dynamic workloads enter the serving runtime, but it
> never requires the runtime to graph them.  Staticity is recovered only for
> templates that remain correct and useful online.

## What Is Still Not Fully Solved

The remaining blocker for a final OSDI/SOSP-level result is trusted same-request
graph-vs-fallback validation inside vLLM.

The current strict runtime cannot safely admit a residual template from a
separate-engine positive observation.  It needs one of:

- an isolated same-engine dry-run context with cloned KV/cache metadata;
- side-buffer graph replay plus output compare before committing user-visible
  state;
- deterministic rollback of all mutated serving state after graph shadow
  replay;
- a weight-sharing shadow engine with provably equivalent mutable state.

Once that exists, the large model matrix becomes meaningful:

- Qwen3-32B, Qwen3-235B, Qwen3.5-35B;
- bs=64 and bs=128;
- vLLM, SGLang, Torch exact-only naive CG, and GraphAdmit strict live runtime;
- metrics: token identity, useful coverage, negative graph rate, avg/P50/P95/P99,
  convergence time, evictions, drift recovery.

## Final Assessment

This round improves correctness more than headline speed.  That is the right
tradeoff for a systems paper: the strongest story is not that GraphAdmit always
beats SGLang/vLLM on every trace today, but that it defines and enforces the
missing correctness boundary for dynamic CUDA Graph serving.

The next implementation milestone is not bucket tuning.  It is trusted
same-engine shadow replay.  With that in place, the old large-model speedups can
be recovered under the stricter admission model instead of relying on offline or
separate-engine evidence.
